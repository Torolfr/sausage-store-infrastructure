Приложение можно построить с использованием традиционной инфраструктуры на стороне заказчика (это может быть «голое железо», виртуализированная среда или даже приватное облако)
и добавленного к ней публичного облачного провайдера, тогда это будет гибридное облако (Hybrid). Также приложение можно развернуть полностью в двух публичных облаках,
и тогда систему в целом можно называть Multicloud Native Service. Рассмотрим возможное построение в Yandex.Cloud. Оба облака можно построить с использованием оркестратора K8S (Managed Service for Kubernetes). 
На представленном рисунке мы видим возможное построение приложения через Multicloud Native Service — кластер из двух облаков, в котором:
1.	Есть внешний Load Balancer для распределения нагрузки между облаками.
2.	Web Serevers — это реплики frontend-сервиса на основе nginx.
3.	Application Servers — это реплики backend-сервиса и backend-report-сервиса.
4.	DB Servers — это базы данных PostgreSQL (Managed Service for PostgreSQL) и MongoDB (Managed Service for MongoDB). Между соответствующими БД настраивается репликация.
Достоинства  данной архитектуры:
·	безопасность;
·	возможность создавать ВМ у разных провайдеров (в разных странах);
·	отказоустойчивость;
·	масштабируемость.
Недостатки данной архитектуры:
·	дороговизна;
·	возможное недоверие клиентов к облаку;
·	сложность архитектуры;
·	возможные требования со стороны провайдеров (особенно, если это связано с законами других стран).
